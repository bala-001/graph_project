## Vision

### 10x Check (post-outside-voice shrink)

Pre-CEO, idea_11.3 committed to building a custom in-house graph engine (PageIndex-equivalent + extracted-fact knowledge graph) as a post-extraction layer.

Mid-CEO (iteration 3 state), the plan accepted tri-architecture B+D+F + OSS + research publication + guardrails — every maximalist candidate on the table.

**Iteration 4 (post-outside-voice, D8 user-accepted)**: the plan shrinks back to a smaller, architecturally honest Phase 1. The outside voice argued — and the user agreed — that the tri-architecture parallel commit was unjustified: if Approach D emits structured edges natively, Approach B is a Phase-2 query layer over D's output, not a parallel architecture; Approach F is a feature, not an architecture; OSS + research publication are resume-building bolted onto a client complaint.

The corrected 10x framing:

1. **Problem-anchored framing (D1, preserved)** — the project commits to fixing the relationship/contradiction problem; graph SHAPE is the leading hypothesis that Phase 0 data validates or replaces.
2. **Phase 1 commits to D + guardrails ONLY** — modify extraction to emit structured edge-triples natively (Approach D) AND add extraction-time consistency check (guardrails as sub-component of D, not separate track). Two engineers / one quarter. Honest scope for the client complaint shape.
3. **Phase 2 is B-as-query-layer-over-D's-output** — once D ships and shadow data confirms the relationship-extraction layer works, Approach B (the in-house structural-index + graph query surface) becomes the natural next layer. B is a query system over D's outputs, not a parallel engine to D.
4. **F (embedding similarity), OSS release, research publication — DEFERRED post-Phase-1** — each has a concrete revisit trigger tied to Phase-1 shadow data. None is rejected outright; all are post-Phase-1 candidates whose value is verifiable only after D ships.

The throughline: the project becomes "PAIQ's relationship-extraction initiative" — a focused, shippable, architecturally honest Phase 1 that fixes the client-complaint shape, with all the maximalist additions preserved as triggered Phase-2 candidates rather than concurrent Phase-1 commitments.

### Why now

Two coherence reasons hold up the tri-architecture decision:

1. **The Phase 0 work is shared across B, D, and F.** The complaint root-cause audit, the boundary-identification rubric, the structure-prototype POC all inform every approach. Running them serially across B-then-D-then-F wastes the audit work; running them together leverages it.
2. **The user's existing IP-ownership stance only pays off if visible.** Custom in-house build is dead weight without external manifestation. Open-source release + research publication ARE the external manifestation of the IP-ownership thesis. Without them, "build it ourselves" is a private cost; with them, it's a public asset.

## Accepted Scope (Phase 1 — POST-OUTSIDE-VOICE SHRINK)

## Eng Review Findings (Iteration 5 — Post `/plan-eng-review`)

Twelve substantive decisions made during eng review on this CEO plan. Each is locked here for downstream implementers.

### Architecture
- **D1 (schema-guided decoding)**: Approach D uses **provider built-in structured outputs** (OpenAI `response_format: json_schema` strict mode OR Anthropic tool-use mode). NOT a custom decoder, NOT a constrained-decoding library, NOT the cascade-OCR Pydantic-retry pattern. **Known limit**: provider built-ins enforce JSON shape, NOT semantic correctness — semantic-correctness defense lives in the eval suite (D6 / D11) + guardrails, not in D1.
- **D2 (guardrails partial-edge state model)**: **Persist-as-you-go** — every emitted edge written to in-document JSON metadata immediately. `extraction_complete: bool` flag distinguishes finished from partial; mutated within a single document's extraction run only; replay journal on crash.
- **D3 (cascade-OCR coupling)**: Cascade-OCR judge **re-calibration** is a D Phase 1 deliverable. Re-run the existing cascade judge calibration on D-mode extraction output BEFORE D ships to production-shadow; confirm judge-pass-rate stays ≥95% on the cascade-OCR eval set. **Re-cal uses existing OCR-quality labels, NOT new relationship-quality labels** (per D12 clarification).

### Code Quality
- **D4 (guardrails FP rate definition)**: Three counters per document tracked: (1) `rejected-then-same-edge` = real FP, (2) `rejected-then-different-edge` = TP, (3) `rejected-then-retry-exhausted` = analyst flag. Kill Criteria "Week-6 guardrails false-positive rate" 1-15% bound applies to counter (1)/total only.
- **D5 (spec polish — Premise 3 + shadow data definition)**: Design doc Premise 3 structure-prototype POC (5-doc kill + 50-doc greenlight + Cohen's kappa ≥ 0.7) is **Phase-2-only** with Approach B. NOT a Phase 1 gate for D + guardrails. Shadow data definition: **D-output vs. analyst-corrected ground truth** on the cascade-OCR eval set. Precision = D ∩ GT / D. Recall = D ∩ GT / GT.

### Tests
- **D6 (complete Phase 1 test coverage)**: All 23 edge-focused tests + 1 mandatory regression = 23 tests. Boil the lake.
- **D11 (regression suite expansion)**: ADD ~10 iso-precision regression tests across ALL existing field types (drug name, age limit, dates, step therapy entries, quantity limit, etc.) — D modifies the same prompt that produces them. Total Phase 1 test count: **~33-35 tests** (23 edge + ~10 existing-field regression). Test plan artifact at `~/.gstack/projects/projects_poc_innovations/2125509-unknown-eng-review-test-plan-20260526-172054.md`.

### Performance
- **D7 (I/O amplification)**: Batched-write journal — buffer flushed every 10 edges OR every 5 seconds (whichever first). Preserves D2 crash-recovery semantics; cuts writes by 10-100x.

### Cross-model integration (iter-5 outside voice findings — user-accepted)
- **D9 (timeline honesty)**: Phase 1 = **2-2.5 quarters** with 2 engineers (was: 1-1.5 quarters). Reflects the critical-path bottleneck (Q3 → F1 → T7 → T8a serial) + cascade re-cal + regression suite + journal/recovery semantics + rollback path. Update sponsor email (T1) to reflect this; portfolio plan's ~12 eng-week budget overlap stays as-is.
- **D10 (multi-turn protocol)**: Extraction is **multi-call** (chunk-by-chunk per document). Guardrails fire **AFTER each LLM call, BEFORE the next chunk's call commits**. Matches D2 + D7 architecture.

### D12 — Five small spec additions (all accepted)

- **FP canonicalization (per D4)**: Edges are canonicalized BEFORE comparison for the same-edge counter. Canonicalization rules: sort tuple elements where order-agnostic, use canonical drug node IDs (resolved through PAIQ's drug dictionary), normalize predicate names (case-fold, strip whitespace), strip trailing punctuation. Without canonicalization, temperature > 0 produces serialization drift that under-counts true FPs.
- **Rollback feature flag (per Week-4 Kill Criteria)**: `paiq.d_extraction.enabled` flag gates the prompt-mode swap. Baseline prompt path preserved in code, NOT deleted. On Week-4 Kill Criteria fire, flip flag false; revert is one config change, not a code rollback.
- **Downstream consumer policy for `extraction_complete=false`**: Downstream consumers (analyst tooling, client-facing outputs, batch reports) treat `extraction_complete=false` rows as **invisible** — equivalent to "this document hasn't been extracted yet." A background GC job sweeps `complete=false` rows older than 24 hours, considering them orphaned-from-crash, and re-queues for re-extraction.
- **Cascade re-cal label scope (per D3)**: Cascade-OCR judge re-calibration uses the existing OCR-quality labels on the cascade-OCR eval set. NO new relationship-quality labels needed for re-cal. The verification is "judge-pass-rate ≥95% holds when its input distribution shifts from baseline-extraction-output to D-mode-extraction-output." Relationship-quality labels (per F6) are for D's own eval suite, not for cascade re-cal.
- **Statistical significance on Kill Criteria gates**: All Phase-2 trigger thresholds (85% precision/recall, etc.) on N=50 shadow docs require **lower-bound of 95% confidence interval ≥ threshold**, NOT just point-estimate ≥ threshold. Standard Wilson or Clopper-Pearson interval applies. Prevents point-estimate noise from triggering kill or pass.
