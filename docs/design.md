## Problem Statement

PAIQ extracts individual criteria facts (drugs, requirements, age limits, step-therapy entries, effective dates) correctly today, but downstream clients are complaining about **contradictory or incomplete criteria outputs** — cases where the individual facts are extracted but the relationships between them are lost or contradictory. Examples include split prerequisite chains (Drug A requires fail-on-B; B has age restrictions — extracted as 3 disconnected facts), policy sections that flow across multiple pages and get partially captured, and contradictions between criteria that the pipeline outputs without flagging.

Idea_11.3 proposes a per-document knowledge graph (Drugs/Indications/Requirements/Conditions/Exceptions as nodes; requires/excludes/applies_to/overrides/effective_from as edges) with query support for transitive relationships and inconsistency detection. Claimed impact: 10-15% improvement in capturing complex multi-step criteria.

**Reshaped scope (user-confirmed during diagnostic)**: The chosen design extends idea_11.3 to TWO graph concepts, both built in-house with no vendor dependency:

1. **Document-structure graph** (PageIndex-equivalent from VectifyAI, but built custom in-house): hierarchical/tree index over the document — sections, headings, page-level features, content flow across pages. Addresses the "policy content flows across multiple pages" reality the user identified.
2. **Extracted-fact knowledge graph** (idea_11.3 as written): nodes for drugs/requirements/etc., edges for relationships. Sits on top of the structure graph and is built from extraction outputs.

Together they form a single custom in-house document graph service.

## Demand Evidence

- **External client complaints (user-confirmed)**: Clients and downstream systems have submitted complaint tickets about contradictory or incomplete criteria outputs. External pressure is the strongest possible demand class — the pain has an external champion, not internal opinion.
- **Multi-page content flow (user-confirmed during 11.2 diagnostic)**: Policy documents have content that flows across multiple pages; this is a known structural problem that today's chunk-based extraction handles poorly.
- **Reference signal**: User specifically referenced PageIndex by VectifyAI as a comparable external service, indicating they have already done landscape research and concluded the pattern is the right shape for the problem.

## Status Quo

- Extraction returns individual facts but does not link them into relationships.
- No per-document knowledge graph or cross-fact consistency check.
- Multi-page policy content is handled chunk-by-chunk; cross-page relationships (e.g., a definition on page 3 modifying a criterion on page 17) are not systematically reconstructed.
- Client complaints are handled reactively — analyst investigates the specific complaint, corrects the output, sends back to client. The pattern is not generalized into prevention.
- No detection of internal logical inconsistencies (circular dependencies, contradictions between criteria for the same drug-indication).

## Target User & Narrowest Wedge

**Target user**: PBM analysts (catch missed relationships during QA today) and PBM clients / downstream consumer systems (the source of the complaint tickets driving demand).

**Chosen wedge (Approach B — Full In-House Custom Build, Phase 0 Gated)**: Build a custom in-house document graph service covering both the document-structure index (PageIndex-equivalent) and the extracted-fact knowledge graph layer. Gated by a 2-3 week Phase 0 that validates the three flagged premises before deep build commits. User explicitly chose this over Approach A (Contradiction-Only Wedge) and explicitly ruled out Approach C (VectifyAI managed service).

## Constraints

- **In-house custom build (strong preference, user-stated; revisitable only via Phase-0 bake-off + sponsor sign-off)**: Strong default toward custom in-house build with no VectifyAI dependency and full IP ownership of the structural index AND the extracted-fact graph. The Phase-0 bake-off has decision power to flip this preference IF VectifyAI quality is comparable on PBM PDFs AND sponsor explicitly signs off on the build-vs-buy reversal. Custom build is justified by IP/vendor/competitive concerns (user has implicit context here).
- **Per-document processing model (user-stated)**: PAIQ processes documents individually; the graph is built per document, not as a cross-document corpus index. This bounds the graph size and the query complexity meaningfully.
- **Intrapreneurship constraint**: Same as 11.1 sibling doc — funding, headcount, and roadmap survive only as long as sponsor priorities align. Reorg risk applies.
- **Production constraint**: PAIQ serves paying clients. The graph layer adds latency and complexity; rollout uses shadow mode in Phase 1 before any client-facing change.
- **Capacity constraint**: 11.1-B (MoE Phase-0-Gated Two-Quarter Rollout) is already committed. Adding 11.3-B in parallel means TWO simultaneous quarter-plus initiatives. Confirm engineering capacity for both or stagger them in the Cost section.

## Premises

Five premises support this plan. Three were flagged shaky by the user.

1. **(SHAKY)** The client complaints are caused by missed relationships at extraction time (where a graph helps), not by upstream extraction errors (where a graph just propagates the wrong facts faster). *Phase-0 mitigation*: audit 30-50 actual complaint tickets, classify root cause (extraction error / missed relationship / contradiction / incomplete coverage). If <50% are relationship/contradiction class, downgrade scope to Approach A (contradiction-only) or pause for re-scoping.
2. **(SHAKY)** Building a PageIndex-equivalent in-house is meaningfully better than using the VectifyAI managed service. *Phase-0 mitigation*: a 1-week build-vs-buy bake-off — run VectifyAI's API against 10 representative PBM documents (under a trial or short eval license), measure structural-indexing quality vs. a minimal custom prototype, decide build-vs-buy on data not assumption. If VectifyAI quality is comparable AND procurement is plausible, the custom-build effort is dead weight; revisit Approach C.
3. **(SHAKY)** PageIndex-equivalent + graph reasoning on multi-page PBM PDFs is a tractable engineering problem within 1-2 quarters, not a research problem. *Phase-0 mitigation, two-stage*: **Stage 3a (5 days)** — structure-prototype POC on 5 representative documents (multi-page, tables, OCR'd, amendments). This is a **kill gate**, not a greenlight: if the 5-doc POC fails the threshold below, abort Approach B immediately. **Stage 3b (2 weeks)** — if 3a passes, scale the prototype to **50+ representative PBM documents** measuring BOTH quality AND throughput / cost per document. Stage 3b is the **greenlight gate** for Phase 1. **Boundary identification = correct identification of: (i) section headings and their hierarchy, (ii) cross-page content continuation (paragraphs/tables that span pages), (iii) table-of-contents → body alignment, (iv) amendment overlay markers**. Rubric: 2 analysts independently label gold-standard boundaries on the POC sample; **Cohen's kappa ≥ 0.7 between analysts** is required for the gold-standard to be usable; prototype output must achieve **≥85% F1 against the gold-standard boundary set**. Throughput floor in 3b: prototype must process a document in ≤ 30 seconds on standard infra.
4. The graph layer (construction + queries) runs within current per-document latency SLA budget. Not flagged shaky by user — preserved as user's confidence.
5. There is an internal architectural / IP / vendor-lock-in reason that justifies not using VectifyAI's API. Not flagged shaky by user — preserved as user's call, same pattern as 11.1 Premise 5 (user-defended).

## Approaches Considered

### Approach A: Contradiction-Only Wedge (cheapest test; not chosen)

Build only the pairwise consistency checker — detects when two extracted facts about the same drug-indication contradict each other. No graph traversal, no document-structure index.

- Effort: S (3-5 weeks).
- Risk: Low.
- Pros: Cheapest demoable artifact; no vendor decision needed; easy rollback; addresses the contradiction subset of client complaints quickly.
- Cons: Doesn't address missed-relationship complaints; doesn't build the structural index that would help multi-page content flow; smaller ROI ceiling.
- Completeness: 6/10.

### Approach B: Full In-House Custom Build (chosen)

Phase 0 (3-5 weeks, hard gate): Complaint root-cause audit + VectifyAI bake-off + structure-prototype POC (5-doc kill gate, then 50-doc greenlight gate per Premise 3 mitigation) + sponsor KPI conversation + sponsor written commitment + SLA budget confirmation. Phase 1 (Quarter 1): Custom PageIndex-equivalent structural index for PBM PDFs + extracted-fact graph data model + basic ingestion + in-document JSON graph storage. Phase 2 (Quarter 2): Query layer (transitive prerequisite resolution, contradiction detection, exception traversal) + shadow rollout. **Analyst-facing visualization explicitly out of scope** for Phase 1/2; deferred to a follow-on doc.

- Effort: XL (1-2 quarters; longer if P3 turns research-flavored).
- Risk: Medium-High after Phase-0 gates; very high without them.
- Pros: Addresses both contradiction and missed-relationship complaints AND multi-page content flow; full IP ownership; reusable across PBM accounts; largest sponsor narrative.
- Cons: Large upfront commitment; P2 and P3 are unproven; runs in parallel with 11.1-B which may stress engineering capacity; production integration is non-trivial.
- Completeness: 9/10 with Phase-0 gates.

### Approach C: Hybrid Build/Buy with VectifyAI (user rejected)

Use VectifyAI managed service for the structural index; build the extracted-fact graph layer in-house. Eliminates P2 and P3 risk by buying the structurally-hardest layer.

- Effort: M-L (6-10 weeks).
- Risk: Low-Medium (vendor and procurement risk only).
- Pros: Fastest path; eliminates engineering risk on structural indexing; quicker time-to-value.
- Cons: Vendor dependency on critical path; procurement / data-residency / BAA at Cognizant scale may take months; IP concern (structural index becomes VectifyAI's, not Cognizant's). User explicitly ruled this out.
- Completeness: 10/10 if vendor risk is acceptable (user says it isn't).

### Phase-1 implementation defaults (set now so engineering is not blocked)

- **Graph storage**: **In-document JSON metadata for Phase 1** (single graph object persisted alongside each document's extraction output, no new database dependency). **Neo4j or dedicated graph DB evaluation deferred to Phase 2** when cross-document queries become relevant. Rationale: per-document graphs are small (single-digit MB) and graph DB infra carries operational cost that's not justified at Phase-1 scope.
- **Complaint root-cause classification rubric**: Four classes, with one-paragraph definitions and worked examples:
  - **Extraction error**: An individual field value extracted incorrectly (wrong drug name, wrong age limit number, etc.). Example: "Drug A age limit extracted as ≥18; document says ≥21."
  - **Missed relationship**: All individual facts extracted correctly but the link between them is lost. Example: "Drug A requires step therapy with B (extracted); B has age restrictions (extracted); pipeline did not surface the chain Drug A → B → age."
  - **Contradiction**: Two extracted facts about the same drug-indication conflict. Example: "Age limit ≥5 AND age limit ≤65 both extracted for same drug-indication, no flag."
  - **Incomplete coverage**: An entire section/page of relevant content was not extracted. Example: "Amendment section on page 17 was skipped; criteria changes from amendment not reflected in output."
  - **Process for the audit**: 2 analysts independently classify a 20% subsample (6-10 tickets); compute inter-analyst agreement (Cohen's kappa target ≥ 0.7); if agreement is below floor, refine the rubric and re-do classification before relying on the 50% threshold.
- **Boundary-identification rubric** for the structure-prototype POC: defined in Premise 3 mitigation above.

## Success Criteria

- **Primary**: Reduction in client complaint tickets attributable to contradiction or missed-relationship root causes by at least 50% in 90 days post Phase-1 shadow launch (subject to sponsor KPI confirmation in Week 1).
- **Secondary**: Phase-1 deliverable produces a per-document graph view (structure + facts + relationships) for at least 90% of PBM documents processed during shadow.
- **Tertiary**: The Phase-0 complaint root-cause audit becomes a reusable internal artifact (PAIQ Client Complaint Taxonomy v1), valuable regardless of full graph outcome.
- **Failure → kill**: See Kill Criteria above.

## Cost

- **Engineering capacity**: Quarter-1 = 1 senior + 1 mid-level engineer dedicated to 11.3 (structure index build is the cost driver). Quarter-2 = same staffing (visualization is explicitly out of Phase 1/2 scope — see Open Questions; no frontend engineer needed for the chosen scope).
- **Capacity-sharing risk with 11.1-B**: 11.1-B's two-quarter commitment uses similar staffing patterns. Running both simultaneously requires roughly 4 engineers total across Q1, plus shared product and ops capacity. If headcount is constrained, sequencing (11.3-B after 11.1-B Phase 1) reduces parallel-team load.
- **Phase 0 cost**: ~3 engineer-weeks (audit + bake-off + structure POC).
- **Inference cost delta**: Graph construction calls happen once per document, mostly post-extraction. Negligible additional LLM cost (a few small LLM calls for fact-extraction-to-graph-node mapping if not deterministic). NO change to vision-model spend, which the user identified as the dominant cost line.
- **Storage cost delta**: Per-document graph metadata stored alongside extraction outputs. Single-digit MB per document typical; negligible at PAIQ volume.
- **VectifyAI trial cost (Phase 0)**: Trial license or pay-per-use eval; budget $500-2000 for the 10-document bake-off.

## Compliance / Data Handling

Same baseline as 11.1 sibling doc — PBM step-therapy and PA criteria data is potentially **PHI-adjacent** under HIPAA. Specific 11.3 considerations:

- **VectifyAI bake-off**: BEFORE running the 10-document bake-off, confirm with Cognizant data governance whether sending PBM documents to a third-party API is permissible. If not, the bake-off uses synthetic / de-identified documents OR Approach C is permanently off the table regardless of bake-off quality.
- **In-house graph storage**: Graph data inherits whatever data classification the extracted facts already have. No new PHI surface introduced.
- **Audit logging**: Graph queries should be logged for the same audit-trail standards as extraction calls. (Analyst-facing visualization is out of Phase 1/2 scope; its compliance question is deferred with the visualization itself to the follow-on doc.)

## Migration Plan (Phase 2)

Phase 2 rollout of the graph layer uses a per-document-type feature flag:

- Flag granularity: `paiq.graph.<feature>.<document_type>` — per-feature (structure-index / fact-graph / contradiction-detection / transitive-queries) × per-document-type (drug-list / policy / amendment / formulary).
- Initial rollout: graph runs in shadow mode for ALL documents in Phase 1; Phase 2 promotes feature-by-feature to analyst-facing then client-facing surfaces.
- Per-feature rollback rule: if graph queries cause >5% latency regression OR contradiction-detection precision drops below 85% on a 5-day rolling window, auto-disable that feature.
- Client communication: graph-derived contradiction flags and transitive-query results are NEW outputs not present in current pipeline; clients receive a change-log entry before any graph-derived content reaches their downstream consumption (owner: client account team).
