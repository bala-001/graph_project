## Affected Pipeline Stages (no UI scope; backend extraction pipeline)

- Extraction prompt path — modified to emit structured edges alongside existing field-level outputs
- New guardrails component sitting at the extraction-emit boundary
- In-document JSON metadata storage — now includes structured edges and an `extraction_complete` flag
- Cascade-OCR LLM-judge — needs re-calibration on D-mode output BEFORE D ships to production-shadow (per D3)
- Shadow comparison harness — D-output vs. analyst-corrected ground truth on the cascade-OCR eval set

## Key Behaviors to Verify

- **Edge extraction precision** per edge type (requires/excludes/applies_to/overrides/effective_from). Each edge type is independently labeled and measured against the analyst-corrected eval set.
- **Baseline regression** — the existing field-level extraction precision (drug name, age limit, step therapy entries, etc.) must NOT regress on a held-out set after D's prompts ship. Mandated test per the eng-review REGRESSION RULE.
- **Guardrails firing** on the four detection scenarios:
  - Circular dependency in prerequisite chains
  - Contradictory age limits on same drug-indication
  - Prerequisite chain mismatch with already-extracted facts
  - Direct contradiction between two edges (same drug-indication, conflicting requirement)
- **Retry semantics** — guardrail rejection → retry with validator error attached → either same edge (counter 1 = FP), different edge (counter 2 = TP), or exhaustion (counter 3 = analyst flag). All three telemetry counters fire correctly.
- **Persist-as-you-go semantics** — every emitted edge written to JSON metadata immediately; `extraction_complete` flag is `false` mid-extraction, set `true` only when the whole document finishes AND all guardrails pass. Downstream consumers respect the flag.
- **Provider structured-output enforcement** — schema violations from the LLM provider's built-in mode are essentially impossible by construction, but the fallback path (rare schema rejection) must surface a clear analyst flag, not a silent drop.
- **Cascade-OCR judge re-calibration** — after D's prompts are finalized, re-run the cascade-OCR judge calibration on D-mode output and confirm judge-pass-rate ≥95% on the same eval set used in the cascade-OCR plan.
- **Shadow comparison precision/recall** — D-output vs. analyst-corrected ground truth: Precision = D ∩ GT / D; Recall = D ∩ GT / GT. The Kill Criteria gate "Quarter-1 D shadow result" depends on these numbers being measurable.

## Edge Cases

- Document with ZERO extractable edges (negative test — D must not hallucinate edges in low-relationship-density documents)
- Document with extraction crash mid-document — `extraction_complete` stays false; downstream consumer must not surface partial state
- Document where every guardrail consultation succeeds — telemetry counter 1/2/3 all stay at zero; verify FP rate metric handles zero-volume case (avoid divide-by-zero)
- Document where retry-exhaust fires — the analyst-flag UX path must work; analyst sees a clear "system needed help on this doc" indication
- Document type currently EXCLUDED from auto-ship eligibility (per the portfolio plan's Q5; this 11.3 plan is separate so the list may or may not apply) — verify D still runs and produces edges even if downstream auto-ship skips
- Long document (50+ pages with multi-page content flow) — guardrails partial-edge state grows but doesn't OOM
- Document with conflicting edges that BOTH should be retained as alternates (e.g., amendment override) — verify guardrails don't reject legitimate "newer-overrides-older" patterns

## Critical Paths

- End-to-end happy path: document → modified extraction → edges emitted → guardrails pass → `extraction_complete=true` → downstream consumer reads → analyst sees coherent output. NO regressions on baseline field extraction.
- Guardrails-fire-then-retry-succeeds path: extraction emits problematic edge → guardrail rejects → LLM retries with validator error → different edge accepted → counter 2 increments → extraction completes cleanly.
- Guardrails-fire-then-retry-exhausts path: extraction emits problematic edge → guardrail rejects → retry produces same problematic edge → counter 1 increments → exhaust threshold reached → counter 3 increments → fall back to analyst flag → analyst sees clear "needs review" indication.
- Cascade-OCR + D coexistence: cascade-OCR judge sees D-mode output → judge-pass-rate stays ≥95% on the re-calibrated baseline → cost-savings claim from cascade-OCR plan stays verifiable.
- Crash recovery: extraction crashes mid-document → next run of the pipeline on that document re-runs from scratch (no partial state honored) → `extraction_complete` semantics enforced by downstream consumers.

## Eval Suite Specification (LLM/prompt change scope)

D modifies extraction prompts. Per CLAUDE.md convention for prompt-touching plans:

- **Eval set**: 50-document labeled set already established by the cascade-OCR plan. EXTENDED with per-doc relationship-type labels and per-pair contradiction-validity labels (per F6 in the CEO plan).
- **Required eval runs**:
  1. Edge extraction precision per edge type (run on every D prompt change)
  2. Baseline field extraction iso-precision (regression test — every D prompt change must show non-regression vs. baseline)
  3. Cascade-OCR judge-pass-rate after D-mode re-calibration (one-time before D ships; re-run if cascade judge prompt changes)
  4. Guardrails detection-rate on synthetic contradictory documents (regression-test that the 4 detection scenarios fire correctly)
- **CI gate**: every PR touching extraction prompts OR guardrails OR the structured-edge schema runs the full eval suite and posts results vs. baseline.
- **Frequency**: per-PR on touched files; per-day rolling on production shadow data.
